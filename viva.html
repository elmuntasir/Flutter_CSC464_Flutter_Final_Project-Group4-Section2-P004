App Architecture & Entry Point

How the app starts

Every Flutter app begins at main() in main.dart. Before anything displays, we call WidgetsFlutterBinding.ensureInitialized() — this hooks Flutter's engine to the device before any plugin (like Firebase) tries to use native code. Then we call Firebase.initializeApp() asynchronously and only then call runApp().

void main() async {
  WidgetsFlutterBinding.ensureInitialized(); // ← MUST be first
  await Firebase.initializeApp(
    options: DefaultFirebaseOptions.currentPlatform,
  );
  runApp(const MyApp()); // inflates the widget tree
}

Folder / Layer structure of your project

Your project follows a clean Repository pattern with 3 layers:

Screens (UI)
→
Repositories (Data Access)
→
Models (Plain Dart objects)
→
Firebase (Cloud)

Screens never talk to Firebase directly — they always go through a Repository. This keeps the code clean and testable. The teacher may ask "why did you separate repositories?" — this is your answer.

⚡ Viva Q&A
Q: What does runApp() do?
A: It takes a widget and makes it the root of the widget tree. Flutter attaches it to the screen, and the framework starts calling build() on it.
⚡ Viva Q&A
Q: Why is WidgetsFlutterBinding.ensureInitialized() needed?
A: Plugins like Firebase use platform channels to talk to Android/iOS native code. That channel must be ready before await calls. If you skip it, the app crashes with "binding not initialized".

StatelessWidget

What it is

A widget that never changes after it's built. It has no internal state. Flutter builds it once (or rebuilds it only if the parent rebuilds it). If the data it depends on never changes during that widget's lifetime, use StatelessWidget.

class HomeScreen extends StatelessWidget {
  const HomeScreen({super.key});

  @override
  Widget build(BuildContext context) {
    return Scaffold(...);
  }
}

In your project: HomeScreen, MyApp, and _SplashScreen are StatelessWidgets. They don't call setState().

app.dart → MyApp home_screen.dart → HomeScreen
⚡ Viva Q&A
Q: HomeScreen reads from Firebase — how can it be Stateless?
A: It uses StreamBuilder inside build(). The widget itself doesn't store anything — it just builds a StreamBuilder and lets that handle reactivity. The UI updates come from the Stream, not from setState().

StatefulWidget

What it is & why you need it

A widget that has mutable state. When setState() is called, Flutter rebuilds only that widget (and its children). It's a two-class system: the Widget class (immutable, holds config) and the State class (mutable, holds data and build()).

class LoginScreen extends StatefulWidget {
  const LoginScreen({super.key});
  @override
  State<LoginScreen> createState() => _LoginScreenState();
}

class _LoginScreenState extends State<LoginScreen> {
  bool _isLoading = false; // mutable state
  bool _obscurePassword = true;

  @override
  Widget build(BuildContext context) { ... }
}

StatefulWidgets in your project

  • LoginScreen — loading state, password visibility, error message, animations
  • RegisterScreen — same + more fields
  • AddExpenseScreen — selected category, date, transaction type
  • ProfileScreen — all profile fields
  • TransactionCard — expand/collapse toggle

What triggers setState()?

  • Toggle password visibility: _obscurePassword = !_obscurePassword
  • After sign-in attempt starts: _isLoading = true
  • Tapping a category tile in AddExpense
  • Tapping the transaction card to expand it
  • Switching between Expense / Income type
⚡ Viva Q&A
Q: Why did you use StatefulWidget for TransactionCard?
A: Because it has a local _isExpanded boolean that controls whether the Edit/Delete buttons show. When the user taps the card, we call setState() to toggle it. This state is local to that card — it doesn't need to be shared — so StatefulWidget is the right choice.

Widget Lifecycle

The lifecycle methods — and exactly where you used them

StatefulWidget States go through a series of lifecycle methods. You used three of them explicitly in your project:

initState()

Called once when the State object is first inserted into the tree. Perfect for one-time setup work: starting animations, pre-filling form fields.

@override
void initState() {
  super.initState(); // ALWAYS call super first
  _animController = AnimationController(...);
  _animController.forward(); // start on entry
}
login_screen.dart add_expense_screen.dart

dispose()

Called when the widget is permanently removed. You MUST dispose controllers here to free memory and prevent memory leaks.

@override
void dispose() {
  _animController.dispose(); // stop ticking
  _emailController.dispose(); // free text memory
  _passwordController.dispose();
  super.dispose(); // ALWAYS call super last
}
login_screen.dart profile_screen.dart

The mounted check — critical pattern in your code

After an await (like a Firebase call), the widget might have been removed from the tree by the time the future resolves (user navigated away). Calling setState() on an unmounted widget throws an error. Your code guards against this:

try {
  await _authService.signIn(...);
} finally {
  if (mounted) setState(() => _isLoading = false);
  // ↑ only update if still in the tree
}
⚡ Viva Q&A
Q: What happens if you forget to call dispose() on a TextEditingController?
A: The controller keeps a listener attached to the widget. After the widget is removed, the controller still exists in memory and keeps listening — this is a memory leak. In long sessions the app uses more and more memory until it crashes or slows down.

Forms & Text Input

The Form + GlobalKey pattern (your most important viva topic)

A Form widget groups multiple TextFormFields together. A GlobalKey<FormState> gives you access to that group from anywhere in the code — specifically to call validate() on ALL fields at once.

// Step 1: create the key (in State class)
final _formKey = GlobalKey<FormState>();

// Step 2: attach to Form widget
Form(
  key: _formKey,
  child: Column(children: [ ... TextFormField(validator: ...) ... ]),
)

// Step 3: trigger all validators on button press
void _signIn() {
  if (!_formKey.currentState!.validate()) return; // stops if any field fails
  // proceed with auth...
}
login_screen.dart register_screen.dart add_expense_screen.dart

TextEditingController

Attached to a TextField/TextFormField to read or pre-fill its value programmatically. In AddExpenseScreen's initState(), you pre-fill fields when editing an existing expense: _nameController.text = e.name. Must be disposed in dispose().

Navigation

How Flutter navigation works — the Navigator Stack

Flutter uses a stack of routes (pages). Pushing adds a page to the top; popping removes the top page. Think of it as a stack of cards — the visible screen is always the top card.

LoginScreen
push →
HomeScreen
push →
AddExpenseScreen
← pop
HomeScreen

Navigator.push()

Adds a new screen on top of the stack. The back button will pop it. Used when going from HomeScreen → AddExpenseScreen or HomeScreen → ProfileScreen.

Navigator.push(
  context,
  MaterialPageRoute(
    builder: (ctx) => const AddExpenseScreen(),
  ),
);

Navigator.pushReplacement()

Replaces the current screen with a new one. The replaced screen is removed from the stack — tapping back won't go back to it. Used when going from LoginScreen → RegisterScreen (you don't want the back button to go to Login from Register).

Navigator.pushAndRemoveUntil()

Pushes a new route and removes all previous routes until a condition is met. Used after successful login — you push HomeScreen and clear everything before it, so pressing back doesn't return to the login form.

Navigator.pushAndRemoveUntil(
  context,
  MaterialPageRoute(builder: (_) => const HomeScreen()),
  (route) => false, // remove ALL previous routes
);

Navigator.pop()

Removes the top screen from the stack. Used after saving an expense (Navigator.pop(context) in _submitData()), after closing a dialog, and in the sign-out confirmation dialog.

⚡ Viva Q&A
Q: What is MaterialPageRoute?
A: It's a route that uses the standard Material Design page transition (slide from right on Android, fade on web). It takes a builder function that returns the widget to display. You can also pass data to the destination screen through the constructor.
⚡ Viva Q&A
Q: How did you pass data to AddExpenseScreen when editing?
A: Through the constructor. AddExpenseScreen has an optional expenseToEdit parameter. When editing, HomeScreen passes the expense object: AddExpenseScreen(expenseToEdit: expense). In initState(), we check if it's non-null and pre-fill the form fields.

StreamBuilder — the heart of your app

What is a Stream?

A Stream is like a pipe that delivers data over time — multiple events, not just one. Firebase Firestore provides real-time streams: every time the database changes, the stream emits new data automatically. Your UI rebuilds with the fresh data instantly, with no refresh button needed.

Firestore DB changes
→ emits event →
Stream
→ triggers →
StreamBuilder rebuilds UI

StreamBuilder anatomy — from your code

StreamBuilder<List<Expense>>(
  stream: repository.getExpenses(), // the Firebase stream
  builder: (context, snapshot) {

    // 1. Still waiting for first data
    if (snapshot.connectionState == ConnectionState.waiting) {
      return CircularProgressIndicator();
    }

    // 2. Error from stream
    if (snapshot.hasError) {
      return Text('Error: ${snapshot.error}');
    }

    // 3. Data arrived — use it
    final expenses = snapshot.data ?? [];
    return ListView(...); // build UI with expenses
  },
)
home_screen.dart — expenses list + auth state in app.dart

Nested StreamBuilders — what you did in HomeScreen

HomeScreen has two streams running simultaneously: one for the user's profile (name, currency) and one for the expenses list. They're nested — the outer one provides currency code, the inner one provides the expenses. When either changes, the relevant part of the UI rebuilds.

⚡ Viva Q&A
Q: What is snapshot.connectionState?
A: It's the current state of the stream connection. The key values are: waiting (stream connected but no data yet — show a loading spinner), active (data is flowing — this is the normal state), done (stream closed — won't emit more data), and none (not connected).
⚡ Viva Q&A
Q: What's the difference between a Stream and a Future?
A: A Future delivers exactly one result then it's done — like an HTTP request. A Stream delivers multiple results over time — like a live feed. Firebase's getExpenses() returns a Stream because the database can change at any moment and we want the UI to update automatically each time.

Q: Why not use StreamBuilder for the profile image?

Animations

Explicit animations — LoginScreen entry animation

You used the full animation system with AnimationController, Tween, and CurvedAnimation. The login form fades in and slides up when the screen opens.

// In State class — uses SingleTickerProviderStateMixin for the vsync
late AnimationController _animController;
late Animation<double> _fadeAnim;
late Animation<Offset> _slideAnim;

// initState:
_animController = AnimationController(
  vsync: this, // "this" works because of the mixin
  duration: const Duration(milliseconds: 700),

_fadeAnim = CurvedAnimation(parent: _animController, curve: Curves.easeOut);
_slideAnim = Tween<Offset>(
  begin: const Offset(0, 0.25), // 25% below final position
  end: Offset.zero,
).animate(_animController);
_animController.forward(); // start playing

// In build:
FadeTransition(
  opacity: _fadeAnim,
  child: SlideTransition(position: _slideAnim, child: ...),
)

The simpler animation approach. You set the end values and Flutter smoothly animates between old and new values automatically. Used for category tile selection (color changes smoothly) and the TransactionCard expand/collapse (shadow changes).

SingleTickerProviderStateMixin

A mixin added to the State class to make it a "vsync provider". The AnimationController needs a vsync to sync with the screen's refresh rate (60fps/120fps). Without this mixin, you can't create an AnimationController in a State class.

login_screen.dart, register_screen.dart
⚡ Viva Q&A
Q: What is the difference between AnimatedContainer and explicit animations with AnimationController?
A: AnimatedContainer (implicit) is automatic — you just change a value and it animates smoothly. No controller needed. AnimationController (explicit) gives full manual control — you decide when it starts, stops, reverses, and you can sync multiple animations together. We used explicit in LoginScreen because we needed two animations (fade + slide) triggered exactly when the screen opens.

Data Models & Classes

Why models exist — separating data from UI

A model is a plain Dart class that represents a piece of data. Your Expense and UserModel classes know how to convert between Dart objects and Firestore maps. The UI never manually handles raw map data — it always works with typed objects.

Expense model

  • fromFirestore(DocumentSnapshot) — converts Firestore data → Dart object
  • toFirestore() — converts Dart object → Map to save to Firestore
  • bool get isIncome — computed getter, not a stored field
  • The type field is either 'income' or 'expense' (String)

UserModel

  • fromMap(Map, id) — converts Firestore data → UserModel
  • toMap() — converts → Map for saving
  • int get age — computed getter that calculates age from birthday
  • Map<String, String> customExpenseCategories — user's custom categories stored in Firestore

factory constructor — Expense.fromFirestore()

A factory constructor is a special constructor that can return an existing instance or do complex setup before creating the object. You used it as the standard pattern for deserializing Firestore data:

factory Expense.fromFirestore(DocumentSnapshot doc) {
  Map<String, dynamic> data = doc.data() as Map<String, dynamic>;
  return Expense(
    id: doc.id, // id comes from the document, not the data
    name: data['name'] ?? '', // null-safe with default
    amount: (data['amount'] ?? 0.0).toDouble(),
    date: (data['date'] as Timestamp).toDate(),
  );
}
⚡ Viva Q&A
Q: Why use ?? '' (null-aware default) when reading from Firestore?
A: Firestore documents might have missing fields — especially older documents created before you added a new field. The ?? operator provides a fallback value if the left side is null, preventing a null crash when building the model.

Async / Await & Error Handling

Why async matters in your app

Any time your app talks to the internet (Firebase) or the file system (saving images), it takes time. Dart is single-threaded — if you block the thread waiting, the UI freezes. async/await lets you "pause" a function without blocking the rest of the app.

// Without async/await — this would FREEZE the UI:
void _signIn() {
  final result = authService.signIn(...); // ← can't do this, Future returned
}

// With async/await — UI stays responsive:
Future<void> _signIn() async {
  setState(() => _isLoading = true); // show spinner
  await authService.signIn(...); // pause HERE, but UI still runs
  setState(() => _isLoading = false); // hide spinner
}

try / catch / finally — your error handling pattern

Used consistently throughout all your async operations. Firebase can throw exceptions (wrong password, network error). You catch FirebaseAuthException specifically to get the error code and show a friendly message.

try {
  await _authService.signIn(email: ..., password: ...);
  // navigate on success
} on FirebaseAuthException catch (e) {
  setState(() => _errorMessage = _mapError(e.code)); // user-friendly message
} finally {
  // always runs — success OR failure
  if (mounted) setState(() => _isLoading = false);
}
⚡ Viva Q&A
Q: What does the finally block do and why is it important?
A: The finally block always runs — whether the try succeeded or an exception was caught. We put _isLoading = false there because we always want to hide the spinner, regardless of what happened. If you put it only in the try block, an exception would leave the spinner spinning forever.

Firebase — What Was Used & Where

📌 Note

A separate Firebase deep-dive note is coming. This section gives you a quick map of which Firebase feature maps to which code, so you can answer "where is X used" questions in the viva.

Firebase Auth

  • Sign up with email/password → RegisterScreen
  • Sign in → LoginScreen → AuthService.signIn()
  • Sign out → ProfileScreen → AuthService.signOut()
  • Auth state stream → app.dart → StreamBuilder on authStateChanges() (auto-route to Home or Login)
  • Current user's UID → FirebaseAuth.instance.currentUser!.uid — used in both repositories to scope all data to the logged-in user

Cloud Firestore

  • Data path: users/{uid}/expenses/{expenseId}
  • CRUD: addExpense(), getExpenses() (real-time stream), updateExpense(), deleteExpense()
  • User profile: users/{uid} — saved with SetOptions(merge: true) to not overwrite fields
  • Timestamp ↔ DateTime conversion in both models
  • All data is user-scoped — no user can read another's data
⚡ Viva Q&A
Q: How does the app know which expenses belong to which user?
A: All expenses are stored under users/{uid}/expenses/ where uid is the Firebase Auth user ID. The repository gets the uid from FirebaseAuth.instance.currentUser!.uid and uses it to build the Firestore path. So each user has their own sub-collection that only they can access.
⚡ Viva Q&A
Q: Why are profile images stored locally instead of Firebase Storage?
A: A deliberate design choice. Firebase Storage requires additional configuration, has a cost, and needs internet to load every time. Storing images locally with getApplicationDocumentsDirectory() means they load instantly offline. The trade-off is they don't sync across devices or survive a reinstall.
